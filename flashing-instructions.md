# Flashing Instructions

This file is the authoritative flashing guide for this repository.

## Preconditions

- Build artifacts must already exist.
- The build must have been verified against `build.yaml` and `build-instructions.md`.
- Do not flash firmware built from mismatched shield or snippet definitions.

## Hard Stop Rules

Do not flash if any of these are true:

1. The build log does not confirm expected `Shield(s): ...` and snippet.
2. The DFU ZIP names do not match the intended target (`left` vs `right`).
3. COM ports are unknown or may belong to different devices.
4. You cannot identify whether the ZIP was generated by the latest build.

## Default Serial Ports On This Machine

- Left bootloader serial: `COM27`
- Right bootloader serial: `COM6`

Verify them before flashing if anything has changed.

## Standard Firmware Flash

Flash both Hillside View halves:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flash-firmware.ps1
```

Override ports if needed:

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flash-firmware.ps1 -LeftComPort COM27 -RightComPort COM6
```

The script flashes these packages:

- `scripts/hillside_view_left-nice_nano_nrf52840_zmk-zmk-dfu.zip`
- `scripts/hillside_view_right-nice_nano_nrf52840_zmk-zmk-dfu.zip`

Before flashing, check ZIP modification times and confirm they were produced by the current build run.

If `--touch 1200` is unreliable on this machine, use `-NoTouch`.

## Settings Reset Flash

Use this only when intentionally clearing persistent settings and split bonds.

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flash-bt-reset.ps1
```

That flashes:

- `scripts/settings_reset-nice_nano-zmk-dfu.zip`

After a settings reset:

1. Let both halves boot the reset image.
2. Then flash the real left and right firmware.
3. Re-pair as needed.

## Bootloader Entry

- Put both halves into upload mode before running the flashing script.
- Confirm the expected COM ports are present.
- If ports differ from defaults, pass them explicitly.

## Post-Flash Validation

After flashing normal firmware, verify all of the following:

1. The display uses the expected current layout/widgets.
2. The current keymap changes are present.
3. Left and right halves reconnect normally.
4. The host receives input from both halves.
5. Any diagnostic change being tested is still present.
6. The displayed layer/status widgets match the current config (not a stale-looking layout).

## If Firmware Appears Old

Check these in order:

1. Was the build produced from the correct full shield list from `build.yaml`?
2. Were the correct DFU ZIPs flashed?
3. Did the flashing script target the correct COM ports?
4. Is the behavior actually caused by persistent settings, requiring a settings reset?

Additional checks:

5. Were there older ZIP files in `scripts/` with similar names that may have been flashed by mistake?
6. Was the keyboard reset fully after flash to ensure new image booted?

Do not assume the keymap is stale until the build and flash inputs have been verified.
